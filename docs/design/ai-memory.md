# AI Memory Model and Prompt Versioning

Status: Draft
Owner: Sol (implementation), Jesse (doc steward)
Related: TRITIUM-PLAN §9 "AI systems and memory", Issues #25, #18
Milestone: Tritium-0 design / Tritium-4 Memory and Growth implementation

## Goals

- Replace the single flat chat history with a layered memory model that
  separates authored personality, durable facts, mid-term summary, and
  short-term conversation.
- Tag every prompt the app sends with an identifiable, versioned prompt
  template so behavior changes are reviewable and reversible.
- Provide explicit "eval seams" — points where prompts and replies can
  be captured, replayed, and asserted on without a live LM Studio.
- Define how the memory system behaves in degraded mode (no LM Studio
  reachable, see Issue #18) without dumping raw error text into the
  user-facing chat surface.
- Keep memory persistence aligned with the persistence design
  (`persistence.md`): same envelope, separate schema id, independent
  migrations.

## Non-Goals

- A vector database or embedding store. DesktopPal's memory is small
  enough to fit in plain JSON; semantic retrieval can be added later
  behind the same `IMemoryStore` interface.
- A general agentic framework. The pal makes short conversational
  responses and reactive comments, not multi-step tool plans.
- Replacing LM Studio. The local-first stance and the chosen endpoint
  (`http://localhost:1234/v1/chat/completions`) stay.
- Cross-pet memory. One pet, one memory store.
- Online prompt analytics. Eval seams produce local fixtures; nothing
  leaves the machine.

## Current State

`DesktopPal/AIService.cs`:

- A single `List<object> _messages` holds the whole conversation,
  initialized with one system prompt that interpolates current stats:

  ```csharp
  string systemPrompt = $"You are a digital pet named {state.Name}. " +
                        $"You are currently at level {state.Level}. " +
                        $"Your hunger is {state.Hunger:F0}%, ...";
  ```

- Trimming is "if more than 10 messages, drop index 1". This randomly
  truncates the conversation and keeps the original (now-stale) system
  prompt forever.
- The system prompt is constructed once per `AIService` lifetime. A pet
  that hatches, levels up, or eats during a session will keep telling
  the model it is at the original stats.
- Vision prompts (in `MainWindow.xaml.cs`) are formatted ad hoc and pushed
  through the same `ChatAsync` channel, polluting the conversation
  history with system-observation messages.
- On any HTTP failure, the user sees `"Hmm... I'm a little fuzzy."`
  with no way to know whether LM Studio is down, the model is wrong, or
  the request was malformed.
- There is no notion of "what the pet remembers about the user" beyond
  the active chat window. Restart loses everything.
- There is no test surface for prompts. Changing personality means
  editing a string and trying it.

## Proposed Design

### Memory layers

Four named layers, ordered from most stable to most volatile:

```
┌──────────────────────────────────────────────────────────────────┐
│ L0 — Persona (authored, versioned, read-only at runtime)         │
│   "You are a small desktop companion named {pet.name}..."        │
│   Tone, voice, hard rules (length, no emojis, no markdown).      │
├──────────────────────────────────────────────────────────────────┤
│ L1 — Long-term traits (slow-moving, persisted)                   │
│   Stable facts: user's preferred name, ongoing interests,        │
│   pet's level/age class, named relationships.                    │
│   Updated at most once per session via summary commits.          │
├──────────────────────────────────────────────────────────────────┤
│ L2 — Mid-term summary (rolling, persisted)                       │
│   ~3-6 sentence rolling recap of recent days/sessions.           │
│   Regenerated by a summarization prompt when L3 overflows.       │
├──────────────────────────────────────────────────────────────────┤
│ L3 — Short-term chat (in-memory, per session)                    │
│   The last N user/assistant turns verbatim (default N = 12).     │
│   When N is exceeded, oldest turns are folded into L2 and        │
│   dropped from L3.                                               │
└──────────────────────────────────────────────────────────────────┘
```

Plus an ephemeral, non-conversational lane:

```
┌──────────────────────────────────────────────────────────────────┐
│ Lx — Stat snapshot (ephemeral, recomputed every prompt)          │
│   Current hunger/hygiene/happiness/energy/level/mood.            │
│   Injected as a structured block, never stored, never replayed.  │
└──────────────────────────────────────────────────────────────────┘
```

Rules:

- L0 is authored content owned by Vex; the runtime treats it as a
  versioned blob it does not edit.
- L1 only grows on explicit "commit" events (session end, milestone).
- L2 is regenerated, not appended. There is exactly one L2 summary at
  any time.
- L3 is the only layer that records verbatim user input.
- Lx is computed at prompt-build time from current `PetStats` so the
  model never sees stale stats.

### Prompt assembly

Every model call goes through a single `IPromptBuilder`:

```csharp
PromptResult BuildChat(ChatRequest req)
{
    return new PromptResult
    {
        TemplateId   = "pet.chat.reply",
        TemplateVer  = 3,
        Messages     = [
            system(L0.Persona),
            system(L1.Render()),
            system(L2.Render()),
            system(Lx.Snapshot(state)),
            ...L3.Tail(N),
            user(req.UserText)
        ],
        Metadata     = new { templateId, templateVer, model, ... }
    };
}
```

Distinct templates exist per intent so they can be versioned and tested
independently:

- `pet.chat.reply` — user-initiated chat.
- `pet.vision.observe` — periodic observation (Issue #18 interplay).
- `pet.letter.read` — letter handling.
- `pet.summary.midterm` — produces L2 from rolled-off L3 turns.
- `pet.summary.longterm` — promotes L2 facts into L1 on session end.

Each template carries:

- a stable string `id`,
- an integer `version`,
- an authored body (in `content/prompts/*.md`, owned by Vex once that
  pipeline exists),
- a list of declared input slots (`pet.name`, `stats.hunger`, etc.),
- a default model and parameter set (temperature, max_tokens).

Outgoing requests log `templateId@version` plus the model name. This is
the basic unit of "what did we send" for prompt review.

### Eval seams

Three seams, all local:

1. **Capture.** Wrapping `IAIClient` in a `RecordingClient` writes
   `{request, response, templateId, templateVer, timestamp}` to
   `%LOCALAPPDATA%\DesktopPal\evals\sessions\<timestamp>.jsonl`.
   Off by default; enabled via a developer flag.

2. **Replay.** A `FileBackedAIClient` reads a captured `.jsonl` and
   returns canned responses by request hash. Used in unit/UI tests so
   "what does the pet say when fed at low energy" is reproducible.

3. **Fixtures.** `tests/prompts/` contains hand-curated input scenarios
   (pet state + memory state + user text) and asserts the produced
   prompt matches a checked-in expected payload. This catches accidental
   prompt drift when refactoring without requiring a live model.

The seams sit at the `IAIClient` and `IPromptBuilder` boundaries; they
do not require network mocks or special infrastructure.

### Memory persistence

Memory persists as a separate file under the save root defined by
`persistence.md`:

```
%LOCALAPPDATA%\DesktopPal\saves\memory.json
                                memory.json.bak
```

Envelope reuses the schema described in `persistence.md`:

```json
{
  "schema": "desktoppal.memory/1",
  "schemaVersion": 1,
  "savedAtUtc": "...",
  "appVersion": "...",
  "data": {
    "longTerm": {
      "userPreferredName": "Scott",
      "interests": ["coding", "tea"],
      "namedRelationships": [],
      "lastUpdatedUtc": "..."
    },
    "midTerm": {
      "summary": "Over the last few sessions, Buddy and Scott...",
      "coversFromUtc": "...",
      "coversToUtc": "...",
      "regeneratedAtUtc": "..."
    },
    "shortTermSeed": []
  }
}
```

L3 is in-memory only by default; `shortTermSeed` allows a small carry-over
across sessions if we decide it improves continuity (open question).
Schema versioning lives on this file independently of `pet.json`.

### Service shape

```
AISubsystem
├── IAIClient              ← talks to LM Studio (or replay/recording wrapper)
├── IPromptBuilder         ← assembles versioned templates
├── IMemoryStore           ← reads/writes layered memory, owns persistence
├── IConversationService   ← user-facing chat orchestration
└── AIStatusReporter       ← exposes "online/offline/degraded" to UI
```

The simulation runtime (`simulation-runtime.md`) treats AI calls as
asynchronous integration; replies arrive on the command bus as
`ApplyAIReply` commands and never mutate state directly.

### Degraded mode (interaction with #18)

Issue #18 covers "AI degraded mode and LM Studio status reporting".
Memory-side responsibilities for that mode:

- `IAIClient` exposes `Status` (`Online | Probing | Offline | Misconfigured`)
  and `AIStatusReporter` surfaces it to the UI as a drawer badge.
- When `Status` is not `Online`, `IConversationService` does **not** call
  the model. It instead returns an authored offline reaction drawn from
  L0 ("offline reactions" pool, owned by Vex) parameterized by current
  Lx mood.
- Offline replies are still recorded in L3 with an `offline: true`
  marker so they don't pollute later summaries (L2 regeneration ignores
  them).
- The user-facing chat surface never shows raw HTTP errors. The drawer
  badge communicates connectivity; the bubble shows in-character
  fallback content.
- Probing is on a backoff (e.g. 5s, 15s, 60s, capped at 5min). On
  recovery the status flips to `Online` and the next user turn goes to
  the model normally.
- Vision and letter handling check `Status` before queueing model calls;
  while offline they enqueue a deferred intent ("react to this letter
  later") that is processed on recovery, with a freshness cap so stale
  intents are dropped.

This split — connectivity in `AIStatusReporter`, content in
`IConversationService` + L0 offline pool — keeps #18 implementation
focused while making the memory system aware of the mode.

## Migration Plan

Four steps, design-then-implement, each shippable.

### Step 1 — Extract IAIClient and IPromptBuilder (no behavior change)

- Move HTTP plumbing from `AIService` into `LmStudioAIClient : IAIClient`.
- Move system-prompt construction into a single-template
  `IPromptBuilder` with `pet.chat.reply@1`. The template still
  interpolates current stats — Lx in spirit, even before layering lands.
- Keep the flat `_messages` list for now; the builder produces it.
- Acceptance: chat behavior is identical; the request payload sent to
  LM Studio is byte-equivalent to today's; `templateId@version` is
  logged.

### Step 2 — Layer the memory model

- Introduce `IMemoryStore` and the L0/L1/L2/L3/Lx layering above.
- Persist memory to `saves\memory.json` using the persistence envelope.
- Replace the "drop index 1" trimming with proper L3→L2 folding.
- L2 regeneration uses `pet.summary.midterm@1`.
- Acceptance: a multi-session test shows that facts established in
  session A are referenced (via L2) in session B; verbatim turns from
  three sessions ago are no longer in the prompt; current stats are
  always fresh.

### Step 3 — Eval seams and prompt fixtures

- Add `RecordingClient` and `FileBackedAIClient` wrappers.
- Add `tests/prompts/` with at least: hatch greeting, hungry reply,
  vision observation while offline, letter reply at high happiness.
- Wire prompt fixtures into the test runner.
- Acceptance: changing a template version without updating fixtures
  fails CI; replay tests pass without LM Studio running.

### Step 4 — Degraded-mode integration with #18

- Introduce `AIStatusReporter` and `Status` plumbing.
- Implement offline reaction pool in L0 (content authored by Vex in a
  follow-up content task).
- Update `IConversationService` to short-circuit on non-Online status.
- Update vision/letter integrations to defer or drop based on status
  and freshness rules.
- Acceptance: with LM Studio stopped, the pet still responds in
  character; the drawer badge reads "offline"; on restart of LM Studio,
  the badge flips to "online" within one probe cycle and normal chat
  resumes.

Steps 1 and 2 land inside Tritium-0/1; Steps 3 and 4 land in Tritium-1/4.

## Risks

- **Summarization quality.** A bad L2 summary becomes the pet's
  long-term "truth". Mitigation: keep L2 short (<= 6 sentences),
  regenerate only when L3 overflows, and authored persona (L0) holds
  hard guardrails so a noisy summary cannot override voice.
- **Prompt template sprawl.** Many templates × many versions × many
  fixtures becomes hard to maintain. Mitigation: keep the template
  inventory small (the five listed above), and prefer parameter slots
  over new template ids.
- **Memory drift across schema versions.** Renaming a long-term trait
  field silently loses data. Mitigation: same migration discipline as
  `persistence.md`; L1 is the layer that most needs migration tests.
- **Degraded mode feels canned.** A small offline reaction pool grows
  stale fast. Mitigation: pool design is Vex's responsibility; system
  side just needs to support pool selection by mood and recent-use
  rotation.
- **Latency regressions.** Building four-layer prompts is more work
  than concatenating a list. Cost is negligible (string formatting),
  but eval-seam logging in production builds could be. Mitigation:
  recording is dev-only.
- **Interplay with #18 timing.** If `AIStatusReporter` is built before
  the layered memory lands, offline replies bypass memory entirely and
  later need re-plumbing. Mitigation: Step 2 lands before Step 4.
- **Over-eager summarization on session end** could thrash L1. Limit
  L1 commits to one per session and only on clean shutdown.

## Open Questions

- Should L3 carry across sessions via `shortTermSeed`, or always start
  empty? Default empty; seed is a future toggle if continuity feels
  weak.
- Where do authored content files (L0 persona, offline reaction pools,
  template bodies) live? Proposed: `content/prompts/` and
  `content/reactions/`, owned by Vex, shipped with the app. Confirm
  during Vex's content-pipeline design.
- Is L1 user-visible? Proposed: yes, a "what Buddy remembers" panel in
  the drawer is a future Tritium-4 feature. Out of scope here.
- Do we need a "forget this" user action that scrubs a fact from L1
  and L2? Strongly yes for trust, but UX is out of scope here. Track
  as a follow-up issue.
- How do we evaluate prompt changes qualitatively? Proposed: a small
  `eval-runner` that takes a fixture directory and prints diffs of
  produced prompts; full output evaluation requires a human or a
  scoring rubric, not part of v1.
- Should vision observations be conversational (in L3) or a separate
  log? Proposed separate, so they don't dominate L2 summaries.
